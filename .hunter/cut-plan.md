# Cut Plan: ReplayFence

> Under deadline pressure, cut breadth first. Never cut credibility.

## Must keep

- Public hosted access or concrete G5 deploy handoff with owner, command, and smoke-test target.
- One complete end-to-end core loop: guest onboarding, first action, Pharos latch, replay rejection, saved capsule, proof/detail, history or verifier return surface.
- Sponsor/domain primitive that makes the product stronger and is visible: Pharos Atlantic tx, chainId 688689, PharosScan link, proof/verifier status.
- Runtime proof: cross-browser Playwright flow, mobile smoke, refresh persistence, second-context verification/history proof, claim matrix, and realness audit.
- Empty, loading, validation, success, proof-unavailable, and recoverable error states for the hero path.

## Can cut now

- `/health` diagnostics page if the same relayer/RPC status appears in the app header.
- Custom action builder beyond the seeded safe action and a small payload mutation demo.
- x402 paid API wrapper, wallet developer mode, analytics cards, team workspace, and batch latch planning.

## Cut last

- `/docs/skill` because Phase 1 rewards reusable Skill quality.
- Capsule verifier and export because they prove return value beyond the single demo.
- Visual proof rail, rejection state, and Pharos explorer/proof screenshots because they carry first-pass judge memory.

## Why cut

Extra breadth is removed so public access, hero path, sponsor mechanism, and proof evidence remain credible. ReplayFence wins by being a narrow Skill that works, not by listing optional surfaces that break under judging pressure.

## Risk if cut

Cutting any must-keep item makes the project look like a guided demo shell: no public URL means judges cannot test it, no Pharos latch means sponsor tech is decorative, no capsule/verifier means unsupported claims, and no runtime proof means the video cannot be trusted.
